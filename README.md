This repository hosts the **HTTP API** and persistence layer of [Picho-RPG](https://picho.org). It pairs with [`picho-rpg-front`](../phantasy-star-rpg.front) (React 19 + Vite). The backend is intentionally minimal — single Fastify process, SQLite on disk, optional Azure Blob storage for uploads — designed to run anywhere from `localhost` to an Azure Container App.

---

## Why this stack?

- **Fastify 5** — fast, schema-first, plugin-driven. We use Zod via `fastify-type-provider-zod` so the same schema validates inputs and types the route handlers.
- **Drizzle + better-sqlite3** — no daemon, no migrations service, no SaaS dependency. The DB is one file under `data/app.db`. Migrations are SQL files generated via `drizzle-kit`.
- **Pluggable storage** — uploads live on local disk by default (`UPLOADS_LOCAL_DIR`) or Azure Blob (`FILE_STORAGE_DRIVER=azure-blob`). The frontend always reads through `/files/*`, so swapping drivers is invisible.
- **JWT-based GM mode** — a single `GM_API_KEY` issues short-lived JWTs after a brute-force-aware unlock. Player and GM endpoints are split via Fastify route plugins.
- **Clean architecture** — `core/` is framework-agnostic (entities + use-cases). `infra/` holds the Fastify glue, Drizzle repositories and storage adapters. `di/` wires everything.

## Features

- ⚔️ **Domain CRUD** for worlds, cities, dungeons, NPCs, monsters, players, quests, sessions, timeline events, lore, wiki, GM notes/images, library documents, character sheets and map markers.
- 🛡️ **GM mode** with brute-force-aware key unlock and JWT-issued sessions.
- 📚 **Library** with format-aware uploads (PDF, DOC/DOCX, XLS/XLSX, PPT/PPTX, EPUB, MOBI, TXT, MD, CSV, ZIP) and **automatic MOBI → EPUB conversion** via Calibre.
- 🗄️ **Pluggable file storage** — local disk or Azure Blob — sharing the same `/files/*` URL contract.
- 🔒 **Security defaults**: Helmet (strict CSP, no script-src), per-IP rate limit, CORS allow-list, MIME allow-list on uploads, 512 KB JSON limit, multipart size cap, JWT auth.
- 📖 **OpenAPI** auto-generated by `@fastify/swagger` + Swagger UI at `/docs`.
- 🧪 **Vitest** integration tests with a real SQLite test DB.
- 🐳 **Containerised** (`Dockerfile`) and deploy-ready for **Azure Container Apps** (manifests under `deploy/azure`).

## Quick start

### Requirements

- Node.js **>= 20**
- npm (or pnpm/yarn — npm is what CI uses)
- (Optional) [Calibre](https://calibre-ebook.com/download) for `.mobi` → `.epub` conversion. The server still works without it; affected uploads are stored as-is and the frontend offers a download instead of a viewer.

### Install & run

```bash
cp .env.example .env       # then edit values
npm install
npm run drizzle:migrate    # apply SQL migrations to data/app.db
npm run dev                # tsx watch on src/app.ts
```

The dev server listens on `http://localhost:3010` by default. Swagger UI: `http://localhost:3010/docs`.

| ------------------------------------ | -------------------------------------------------------- |
| `npm run dev`                        | `tsx watch src/app.ts` — restarts on file changes.       |
| `npm run build`                      | TypeScript build to `dist/` + import path fix.           |
| `npm start`                          | Run the compiled `dist/app.js`.                          |
| `npm run drizzle:generate`           | Generate a new SQL migration from the schema.            |
| `npm run drizzle:migrate`            | Apply pending migrations to the configured database.     |
| `npm run lint` / `npm run lint:fix`  | ESLint over the project.                                 |
| `npm run format`                     | Prettier on the whole repo.                              |
| `npm test` / `npm run test:run`      | Vitest in watch / single-run mode (uses `data/test.db`). |
| `npm run test:cov`                   | Vitest with coverage (V8).                               |
| `npm run check`                      | Lint + build + test (used in CI).                        |
| `npm run storage:migrate:azure-blob` | Move existing local uploads to Azure Blob (one-shot).    |

## Storage

The backend supports two upload backends with the same public URL contract (`/files/<key>`):

### Local disk (default)

```env
FILE_STORAGE_DRIVER=local
UPLOADS_LOCAL_DIR=./data/uploads
```

Files land under `UPLOADS_LOCAL_DIR`. The directory is created automatically on boot.

### Azure Blob

```env
FILE_STORAGE_DRIVER=azure-blob
AZURE_STORAGE_CONNECTION_STRING=DefaultEndpointsProtocol=...
AZURE_STORAGE_CONTAINER=uploads
AZURE_STORAGE_ACCESS_TIER=Cool
```

The frontend never sees the difference — it keeps requesting `/files/<key>` and the backend streams from the chosen driver.

To migrate existing local uploads to Azure Blob:

```bash
npm run storage:migrate:azure-blob
```

## 🔑 GM key

Picho-RPG has no user accounts. The split is simple: **everyone is a player by default**, and a single shared **GM key** unlocks the GM panel. There's one GM key per deployment.

### How to set it up

1. **Generate a strong key** (≥ 24 chars, mixing upper, lower, digits and symbols):

   ```bash
   # Linux / macOS / WSL
   openssl rand -base64 48

   # PowerShell
   [Convert]::ToBase64String([byte[]](1..48 | ForEach-Object { Get-Random -Maximum 256 }))
   ```

2. **Set it in the backend `.env`**:

   ```env
   GM_API_KEY=the-generated-string-here
   JWT_SECRET=another-long-random-string-here
   JWT_EXPIRES_IN=8h
   ```

3. **Restart the backend.** On boot, the auth plugin validates `GM_API_KEY`:
   - It must be at least **24 characters**.
   - It must contain at least **3 of these 4 classes**: uppercase, lowercase, digits, symbols.
   - In `NODE_ENV=production`, a weak key (or no key) **aborts boot**. In dev, it logs a warning so you can still iterate.

4. **Hand the key to the GM** out-of-band (DM, password manager, sealed envelope — your call). It is not exchanged anywhere else after the first unlock.

### How players / GM use it

The flow on the frontend:

1. The GM clicks the GM toggle in the settings dropdown and pastes the key.
2. The frontend sends `POST /api/auth/login` with `{ "apiKey": "<the-key>" }`.
3. The backend compares it with `safeCompare()` (constant-time) against `GM_API_KEY`, throttles per IP via the brute-force tracker, and on success issues a JWT signed with `JWT_SECRET`.
4. The frontend stores the JWT and sends it as `Authorization: Bearer <jwt>` on every subsequent request.
5. GM-only routes (e.g. uploads, hidden entity edits, GM notes) check the JWT via the `requireGM` pre-handler.

**Public API:**

```http
POST /api/auth/login
Content-Type: application/json

{ "apiKey": "the-gm-key" }
```

Successful response:

```json
{
  "token": "eyJhbGciOi…",
  "expiresAt": "2026-04-29T03:14:15.000Z"
}
```

Failures:

- `401 Unauthorized` — wrong key. The brute-force tracker records the failure.
- `429 Too Many Requests` — IP is in lockout. The response includes a `Retry-After` header and a `retryAfterSeconds` field.

### Brute-force protection

`src/infra/security/brute-force.ts` keeps an in-memory tracker per IP. After consecutive failures the tracker locks the IP for a backoff period; successful unlocks clear the counter. The state is **per-process** — if you scale horizontally, swap the implementation for a shared store (Redis, Azure Table) before exposing the API publicly.

### Rotation

To rotate the GM key, replace `GM_API_KEY` in the deploy environment and restart the backend. All in-flight JWTs remain valid until expiry; if you need to invalidate them immediately too, also rotate `JWT_SECRET` (this kicks every authenticated GM out).

### Things to avoid

- **Don't commit `.env`.** The repository's `.gitignore` excludes it. The `.env.example` lists every variable.
- **Don't put the GM key in client code.** It only lives in the backend env. The browser only ever sees the JWT.
- **Don't use the same value for `GM_API_KEY` and `CLIENT_SECRET`.** The first authenticates the GM; the second is a shared sanity check that the request comes from a Picho-RPG client.
- **Don't skip the JWT.** Every GM request must include `Authorization: Bearer <jwt>`. The unlock endpoint is the only one that accepts the raw key.

## Calibre (MOBI → EPUB)

When a `.mobi` file is uploaded to the Library, the backend tries to convert it to EPUB using Calibre's `ebook-convert` CLI so the in-browser viewer can render it. If Calibre is not installed, the file is stored as-is and users fall back to downloading it.

<details>
<summary><strong>Install Calibre on Windows</strong></summary>

1. Download and install from <https://calibre-ebook.com/download_windows>.
2. Add the install directory to the system `PATH` (default `C:\Program Files\Calibre2\`).
3. Restart your terminal and verify:
   ```bash
   ebook-convert --version
   ```

</details>

<details>
<summary><strong>Install Calibre on Linux (Debian/Ubuntu)</strong></summary>

```bash
sudo apt install calibre
```

</details>

<details>
<summary><strong>Install Calibre on macOS</strong></summary>

```bash
brew install calibre
```

</details>

## Architecture highlights

- **Hexagonal layering.** `core/` defines what the app does; `infra/` defines how. The DI container in `src/di/container.ts` wires concrete implementations to the use-cases.
- **One controller per domain** under `src/infra/http/controllers/`, paired with a route declaration that wires Zod schemas in `src/infra/http/routes/`.
- **GM gate.** GM-only endpoints register the `auth` plugin; player endpoints don't. The unlock endpoint accepts a key, validates it against the hash of `GM_API_KEY`, applies a brute-force tracker, and issues a JWT.
- **Visibility filter.** A Fastify plugin trims hidden entities from listing/get responses for non-GM callers, so the same endpoint can serve both audiences without leaking.
- **Schema-first.** Every route declares a Zod schema; the type provider gives you `req.body`/`req.query`/`req.params` typed for free, and `@fastify/swagger` builds the OpenAPI spec from the same source.
- **Multipart uploads** go through `@fastify/multipart`, with a per-route MIME allow-list and a global size cap (`MAX_UPLOAD_MB`).

For deeper guidance — repository conventions, where to add a new entity, how Calibre fits in — see [`AGENTS.md`](AGENTS.md).

## Deployment

The repo ships a multi-stage `Dockerfile` and an Azure Container Apps deploy pipeline (`deploy/azure/`).

```bash
# Build the image locally
docker build -t picho-rpg-backend .

# Run with a mounted data dir
docker run -p 3010:3010 \
  -e GM_API_KEY=... -e JWT_SECRET=... -e CLIENT_SECRET=... \
  -v $(pwd)/data:/app/data \
  picho-rpg-backend
```

For Azure Container Apps:

- Logs are pushed to stdout (Pino JSON in prod).
- Persistent disk goes via Azure Files volume mounted at `/app/data`.
- Set `FILE_STORAGE_DRIVER=azure-blob` to push uploads to Blob instead of the volume.

### Production CD

[`deploy-production.yml`](.github/workflows/deploy-production.yml) deploys every push to `main` to Azure Container Apps:

- Azure login uses GitHub OIDC with an Azure identity federated to this repository's `main` branch.
- The workflow builds and pushes an image tagged with `${GITHUB_SHA}` to Azure Container Registry.
- The workflow updates the configured Azure Container App.
- Secrets remain in Azure Container Apps and are referenced as `secretref:*`; secret values are never stored in the repository.

Required GitHub Actions secrets:
